# Phase 5.5 Build Notes — Multi-Connect Server-Side Dedup

## Design

When a client connects to two servers simultaneously and transmits, both servers receive the same DMRD packets with the same `stream_id` (set by the radio hardware). Each server then broadcasts the stream over the cluster bus. Without dedup, local repeaters on each server would hear the transmission twice — once from the local stream, once from the virtual stream.

## Dedup Strategy

Two dedup points handle both race conditions:

### 1. Virtual stream suppression (normal case)
In `_handle_virtual_stream_start()`: before creating a virtual stream, check if the same `stream_id` is already active on any local repeater via `_is_stream_active_locally()`. If so, suppress the virtual stream entirely — the local stream already handles forwarding.

Subsequent `stream_data` packets for the suppressed virtual stream are silently dropped (no key in `_virtual_streams`).

### 2. Stale virtual cleanup (race condition)
In `_handle_stream_start()`: when a local stream starts, scan `_virtual_streams` for any matching `stream_id + slot` and remove them. This handles the rare case where the cluster bus delivers `stream_start` faster than the local UDP path.

## Why stream_id works for dedup

The DMR `stream_id` is a 4-byte identifier at DMRD bytes 16-20, generated by the radio for each PTT. When the same radio transmits to two servers simultaneously, both servers see the **exact same stream_id** — it's part of the DMR air interface, not generated by the server.

## Tests (6 new)
- Virtual stream suppressed when local stream active (same stream_id)
- Virtual stream created when no local match
- Virtual stream not suppressed for different stream_id
- `_is_stream_active_locally`: no repeaters, active stream, ended stream
